# Agent System Checklist（适配 Leo 当前 OpenClaw 环境）

> 来自对 Claude Code / agent system 工程经验的落地提炼。目标不是“更聪明”，而是“更清楚、更稳定、更可验证”。

## 一、Context Surface（上下文表面）

### 核心原则
- 上下文不是越大越好，而是越干净越好。
- 主上下文只保留高价值、常用、会持续影响判断的内容。
- 项目型、一次性、长篇内容尽量按需读取。

### 适合常驻
- `SOUL.md`
- `USER.md`
- `AGENTS.md`
- 最近 1-2 天 `memory/YYYY-MM-DD.md`
- 主会话才读 `MEMORY.md`
- 当前任务摘要（来自 `SESSION-STATE.md`）

### 不适合常驻
- 长篇参考文档
- 旧项目全过程记录
- 大段工具输出
- 低频 Skill 的完整 supporting docs
- 任何“也许会用到”的大块背景

### 实操规则
- 新任务先问：这份信息是“持续影响判断”，还是“只在此任务阶段有用”？
- 后者放 `references/` 或项目目录里，需要时再读。

---

## 二、Rule Surface（规则表面）

### 核心原则
- `AGENTS.md` 是操作系统内核，不是资料库。
- 只写：规则、触发条件、边界、验证标准。
- 不写：长篇背景、项目细节、一次性策略。

### AGENTS.md 里应该有
- 安全红线
- WAL / Working Buffer / VBR
- 什么时候必须确认用户
- 什么时候必须记忆落盘
- 什么时候必须验证

### 不应该有
- 某个具体项目的完整设计过程
- 某次调研的全部结论
- 大量低频技能说明
- 会快速过时的长文档内容

---

## 三、State Surface（状态表面）

### 核心原则
- 长任务必须有显式状态，不要靠“上下文应该还记得”。

### 推荐文件分工
- `SESSION-STATE.md`：当前任务状态 / 决策 / 验证状态 / 下一步
- `memory/working-buffer.md`：上下文超过 60% 之后的危险区对话缓冲
- `memory/YYYY-MM-DD.md`：每日事实记录

### SESSION-STATE.md 至少包含
- Current Focus
- Recent Decisions (WAL)
- Active Constraints
- Next Actions
- Validation Status（建议后续逐步补）

### 触发场景
- 任务持续超过 30 分钟
- 任务分多个阶段
- 配置/部署/代码修改较多
- 需要未来某次接着做

---

## 四、Skill Surface（技能表面）

### 核心原则
- Skill 是“按需加载的工作流”，不是知识堆积。

### 一个 Skill 至少回答 4 个问题
1. 什么时候用
2. 怎么用
3. 输入是什么
4. 什么时候停

### 低频高风险 Skill
默认建议：
- 不自动触发
- 只手动调用
- 明确副作用和停止条件

### 反模式
- description 太泛：如“help with backend”
- SKILL.md 太长：把一切都堆进去
- 只有理念，没有步骤
- 没有输入输出和停止条件

---

## 五、Tool Surface（工具表面）

### 核心原则
- 工具要为 Agent 设计，不要仅为人类设计。

### 好工具特征
- 名字明确
- 作用单一
- 参数可理解
- 错误信息可修复
- 输出规模可控制

### 命名建议
优先：
- `wal-log`
- `dashboard-screenshot`
- `nightly-security-audit`
- `repo-safe-backup`

避免：
- `run`
- `query`
- `do`
- `handle`

### 设计建议
- 大输出尽量支持精简格式
- 错误信息说明“怎么修”
- 能高层封装时，不强迫模型拼底层步骤

---

## 六、Hook Surface（钩子表面）

### 核心原则
- 硬规则不要靠模型记住，尽量自动化。

### 适合 Hook / 自动检查
- 编辑后语法检查
- 配置改动后跑 `openclaw security audit`
- 高风险文件修改提醒
- 受保护路径禁止直改
- 安全巡检自动化

### 不适合 Hook
- 大量开放式推理
- 调研类任务
- 创作类任务
- 高歧义判断

### 判断准则
如果事情满足以下 3 条，优先 Hook 化：
1. 规则清楚
2. 自动可检
3. 做错代价高

---

## 七、Isolation Surface（隔离表面）

### 核心原则
- Subagent 的首要价值是隔离上下文污染，不是并行炫技。

### 适合 subagent 的任务
- 大量网页/资料搜索
- 长文整理
- 代码库扫描
- 大量中间输出
- 审查 / 巡检 / 测试

### 不适合 subagent 的任务
- 简单判断
- 很短的来回讨论
- 强依赖主线程上下文的小任务
- 高频即时问答

### 实操判断
问自己：
> 这件事如果放在主会话里，会不会制造很多噪音？
如果会，就适合隔离。

---

## 八、Compaction Surface（压缩表面）

### 核心原则
- 默认 compaction 不一定知道你最在意什么。
- 重要信息要在压缩前自己落盘。

### 默认容易丢掉的东西
- 架构决策原因
- 哪些方案试过
- 当前验证状态
- TODO / 回滚提示
- 中间结论

### 最低落地方案
- 60%+ 上下文时启用 `working-buffer`
- 长任务持续更新 `SESSION-STATE.md`
- 必要时手写 handoff 摘要

---

## 九、Verification Surface（验证表面）

### 核心原则
- “改了”不等于“完成了”。
- 验证层必须独立存在。

### 最低验证动作示例
- 命令类：关键输出回显
- 页面类：截图 / 可访问性检查
- 消息类：`ok: true` / messageId
- 配置类：重启后跑状态检查
- GitHub：确认 commit / push / 远端存在

### 反模式
- 改文案当改机制
- 提交文件当功能完成
- 没实跑就说修好了

---

## 十、System Design Priority（系统设计优先级）

### 核心原则
后续优化优先级：
1. 边界更清楚
2. 结构更稳定
3. 验证更强
4. 成本更低
5. 最后才是新增功能

### 不推荐的方向
- 为了“显得更聪明”而持续加复杂度
- 不验证就增加新层
- 主上下文越塞越胖
- Skill / Tool / Hook / Subagent 边界混乱

### 推荐的方向
- 高频规则进入内核
- 低频细节进入 supporting docs
- 长任务结构化交接
- 关键步骤显式验证

---

## 一页纸版（最小执行版）

1. 主上下文只放最贵的信息
2. `AGENTS.md` 只写规则，不写项目细节
3. 关键变化先写 WAL，再回复
4. 长任务必须写 `SESSION-STATE.md`
5. Skill 设计重点是触发条件
6. 工具命名要让 Agent 一眼看懂
7. Hook 管硬规则，Subagent 管隔离
8. 压缩前先留交接信息
9. 没验证，就别说完成
10. 优先优化系统边界，而不是继续堆功能
